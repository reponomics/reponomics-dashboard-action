# Post-Mortem: Failed reponomics-dashboard-v0.12.0 Template Publication

Date: 2026-06-20

## Summary

The `reponomics-dashboard-v0.12.0` template release failed after creating the immutable source template tag in `reponomics-dashboard-action`, but before publishing the generated template commit, generated repository tag, and GitHub Release in `reponomics/reponomics-dashboard`.

The failure happened because `scripts/publish_generated_repo.py` treated a missing generated-repository release tag as a fatal `git ls-remote` error. For a first publication attempt, that tag is expected not to exist.

This happened before the project was fully public, which made it the right time to harden the release protocol and clarify the immutable provenance rule.

## Impact

- Source tag `reponomics-dashboard-v0.12.0` exists in `reponomics-dashboard-action`.
- Generated repo release/tag `reponomics-dashboard-v0.12.0` was not created.
- The release train is incomplete for template version `0.12.0`.
- No public generated-template release was published for `0.12.0`.
- The correct response is to leave `0.12.0` as a failed publication and release over it with a new source SHA and new template version.

## Root Cause

The publication script queried the generated repo for both:

```text
refs/tags/reponomics-dashboard-v0.12.0
refs/tags/reponomics-dashboard-v0.12.0^{}
```

GitHub returned:

```text
fatal: couldn't find remote ref refs/tags/reponomics-dashboard-v0.12.0
```

The script did not distinguish "tag absent" from a real git failure. It should have interpreted this as the normal first-publication path and continued to create the generated publication commit and tag.

## Contributing Factors

- Local tests used bare/path remotes, which did not reproduce GitHub's missing remote-ref behavior.
- The release workflow correctly created the source tag before publication, but the publication script failed immediately afterward.
- We initially considered repairing the failed template tag with later tooling, but that would weaken the provenance story.

## Correct Release Rule

Template publication is provenance-bound:

```text
source SHA -> generated template payload -> generated repo commit/tag/release
```

If an immutable source tag is created but publication fails before the generated repo tag/release exists, do not amend, reuse, or quietly repair that version from a later source SHA.

Instead:

```text
failed immutable tag remains failed
next attempt gets a new source SHA and new template version
```

## Corrective Actions

- Fix `publish_generated_repo.py` to treat missing generated tags as absent, not fatal.
- Add regression coverage for GitHub's missing remote-ref behavior.
- Merge the fix normally.
- Release over `0.12.0` with a new source release and next template version.
- Document the rule: failed immutable publication boundaries are superseded, not repaired in place.

## What Went Well

- The workflow failed before publishing a partial generated repo release.
- The source tag makes the failed boundary auditable.
- The issue happened before full public launch.
- The failure clarified the release model: provenance correctness matters more than conserving version numbers.

## What To Improve

- Add a release runbook section for "source tag created, generated publication failed."
- Make tests cover GitHub remote behavior, not only local bare remotes.
- Keep distinguishing action compatibility, template publication, and generated repo provenance as separate concerns.
- Avoid using "repair" language for immutable release artifacts unless the repair runs from the exact same source boundary and preserves the full provenance chain.
